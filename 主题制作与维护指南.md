# Codex Dream Skin 主题制作与维护指南

> 本文是主题作者、视觉设计者和 Release 维护者的统一工作手册。普通安装与日常使用请看 [README.md](README.md)；运行时验证记录和截图清单保留在 `engine/references/`。

> 启动器 Release 不再内置主题。Angelina 浅色/深色主题及其素材来源位于 [Codex Angelina Themes](https://github.com/bilbillm/Codex-Angelina-Themes)；通过控制台导入或复制到本地 `themes/` 目录使用。

## 1. 先理解系统边界

Codex Dream Skin 不是修改 Codex 安装包的主题插件。它通过本机回环地址的 Chromium DevTools Protocol（CDP），把 CSS、图片和少量装饰 DOM 注入到正在运行的 Codex 渲染页面。

| 层级 | 位置 | 职责 | 修改时机 |
|---|---|---|---|
| Release 交互层 | `Codex自定义主题.exe`、快捷方式脚本 | 主题预览、导入、热切换和一键启动 | GUI 行为或安装入口变更 |
| 主题数据层 | `themes/<主题名>/` | 每个可分发主题的图片和 `theme.json` | 新主题、图片或元数据变更 |
| 注入引擎 | `engine/scripts/` | Windows 生命周期、主题校验、CDP 注入和 watcher | 运行机制、安全或兼容性变更 |
| 渲染层 | `engine/assets/renderer-inject.js`、`dream-skin.css` | DOM 分类、视觉样式、视差和装饰 chrome | 布局、交互、动效或 Codex DOM 适配 |
| 可变运行状态 | `%LOCALAPPDATA%\CodexDreamSkin` | 当前主题、暂停标记、watcher 身份和日志 | 不应提交到仓库 |

不要编辑 `WindowsApps`、`app.asar` 或 `.exe`。这个 Release checkout 不含 `src/CodexThemeSwitcher/` GUI 源码；需要修改 GUI 窗口、预览或 `--launch` 的 C# 行为时，必须回到完整源码 checkout 重建。

## 2. 设计规范

### 2.1 设计目标

- **内容优先。** 背景和光效服务于项目选择、提示词输入与对话阅读，不能压过正文或控件状态。
- **画面有安全区。** 首页主要文案和建议卡通常位于左侧；人物、强对比物体和文字不要侵入该区域。右侧安全区主题反过来安排。
- **首页与任务页分工。** 首页可有更完整的主视觉；任务页必须降低细节、对比和局部锐度，避免干扰长对话。
- **浅色与深色分别设计。** 不要只靠滤镜把一张图翻转成另一套主题。确保正文、图标、边框和焦点态在两种外观中都清晰。
- **先统一，再加装饰。** 每个主题先确定主画面、纸面/玻璃表面、文字、强调色和状态色；随后才考虑动效、纹理或装饰标签。

### 2.2 首页画面与排版

| 要素 | 建议 | 避免 |
|---|---|---|
| 主图构图 | 16:9；主体放在与 `safeArea` 相反的一侧；给输入区与建议卡留出低细节区域 | 主体脸部、亮光或大字压在输入框后面 |
| 明暗关系 | 文案区与背景保留明显明度差；通过 hero 遮罩而不是大面积不透明面板保可读性 | 用深重蒙版完全盖住图片，或让浅色正文落在亮背景上 |
| 栅格/纹理 | 低对比、静态、仅作空间感 | 高频噪点、闪烁纹理或肉眼可见的滚动纹理 |
| 玻璃表面 | 边界、阴影和透明度由同一套 token 控制 | 每个卡片各自使用不同的模糊、圆角和阴影 |
| 强调色 | 用于焦点、选中、主要操作和少量信号点 | 把强调色当作大面积背景，或同时引入多个竞争色 |

### 2.3 动效与光效

当前 Angelina 变体已有首页文案/卡片入场、轻量信号反馈、鼠标视差和 `prefers-reduced-motion` 兜底。因此新增动效应遵循以下预算：

- 一屏最多有 **1–2 个注意力焦点** 在动；首屏装饰性入场结束后必须静止。
- 控件反馈使用 `150–300ms`，页面文案或卡片入场使用 `400–700ms`；一次性光带可以延长到 `1.2s`。
- 使用 `ease-out` 进入、`ease-in` 离开；不对工具界面使用循环弹跳、文字打字机或持续镜头推拉。
- 连续动画只用于真实的加载/处理中状态。背景粒子、扫描线和呼吸光都应改为一次性进入效果或交互触发效果。
- 优先修改 `transform` 与 `opacity`，不要持续改变整窗 `background-position`、`filter` 或布局尺寸。
- 所有新动画都必须在 `@media (prefers-reduced-motion: reduce)` 下停止或瞬时完成，不能依赖鼠标移动才可用。

一个克制的首屏方案是：进入时让人物一侧有一次低透明度的暖白到信号红掠光，随后静止；输入框获得焦点时再给边缘一个短促的强调色柔光。它既不会和现有视差竞争，也能把“准备开始工作”的状态说清楚。

### 2.4 可读性和交互检查

- 正文与其表面的对比度至少达到 `4.5:1`；次要文字、边框和图标在浅/深主题均应可辨。
- 不依赖颜色单独传达暂停、错误或选中；保留图标、文字或形状差异。
- Hover、focus 与 disabled 状态不能改变控件布局；键盘焦点必须可见。
- 装饰层必须 `pointer-events: none`，不能挡住输入、滚动、拖拽或原生窗口控件。
- 首屏、任务页、窄窗口和高 DPI 都要检查裁切。视觉修改至少保留一张浅色和一张深色截图。

## 3. 主题包规范

### 3.1 目录结构

一个可扫描主题必须直接位于 `themes/` 下，并在主题目录顶层放置 `theme.json`：

```text
themes/
└─ My Theme/
   ├─ theme.json
   ├─ home.png
   ├─ task.jpg                 # 可选
   ├─ parallax-background.png  # 可选，与前景成对
   └─ parallax-foreground.png  # 可选，与背景成对
```

外部主题应使用 GUI 的“导入主题”复制进主题库；不要让 `theme.json` 的图片路径指向其他磁盘、网络地址、父目录、符号链接或 junction。

### 3.2 推荐的最小主题

```json
{
  "schemaVersion": 1,
  "id": "my-theme",
  "name": "My Theme",
  "image": "home.png",
  "appearance": "dark",
  "art": {
    "focusX": 0.68,
    "focusY": 0.42,
    "safeArea": "left",
    "taskMode": "ambient",
    "parallax": false
  },
  "palette": {
    "accent": "#c85b55"
  }
}
```

### 3.3 运行时字段

| 字段 | 是否必需 | 可用值/格式 | 作用 |
|---|---|---|---|
| `schemaVersion` | 推荐 | 当前写 `1` | 格式声明和兼容性标记 |
| `id` | 推荐且应稳定 | 最多 80 个字符的单行文本 | 主题库身份；已发布主题不要复用或随意改名 |
| `name` | 推荐 | 最多 120 个字符的单行文本 | GUI 中显示的主题名称 |
| `variant` | 可选 | 当前仅 `angelina` | 启用 Angelina 专用渲染器 CSS 与装饰 chrome；省略时使用通用 Dream Skin 行为 |
| `image` | 是 | 主题目录内的相对图片路径 | 首页主视觉，也是缺少任务图时的后备图片 |
| `taskImage` | 可选 | 主题目录内的相对图片路径 | 对话/任务页背景图 |
| `appearance` | 可选 | `light`、`dark`、`auto` | 根 class 与配色模式 |
| `art.focusX` / `art.focusY` | 可选 | `0` 到 `1` | 主图的视觉焦点坐标 |
| `art.safeArea` | 可选 | `auto`、`left`、`right`、`center`、`none` | 文案可读区域与 hero 遮罩方向 |
| `art.taskMode` | 可选 | `auto`、`ambient`、`banner`、`off` | 任务页图片使用方式；`off` 不显示任务页艺术图 |
| `art.parallax` | 可选 | `true` / `false` | 开启时要求同时提供两张视差图 |
| `backgroundImage` | 视差时必需 | 相对图片路径 | 视差背景层 |
| `foregroundImage` | 视差时必需 | 相对图片路径 | 视差前景层 |
| `palette.accent` | 可选 | `#RGB`、`#RRGGBB`、CSS `rgb()`、`hsl()`、`oklch()` 等 | 主强调色；用于注入的 CSS 变量 |

`art.parallax` 为 `true` 时，`backgroundImage` 和 `foregroundImage` 必须成对出现；仅填写图片而不启用该开关不会开启指针视差。

### 3.4 元数据字段的现实边界

已有主题中的 `skinRevision`、`visualRevision`、`brandSubtitle`、`tagline`、`projectPrefix`、`projectLabel`、`statusText` 和 `quote` 可以保留为版本/策展说明，但当前注入器不会将它们传给运行时主题对象，Angelina 的装饰文案仍由 `renderer-inject.js` 固定生成。

因此：

- 只改这些字段不会改变现有 UI。
- 若要让不同主题显示不同的 kicker、引语或项目标签，需要同时修改 `engine/scripts/injector.mjs` 的主题白名单、`engine/assets/renderer-inject.js` 的 DOM 模板、`engine/assets/dream-skin.css` 的样式与相关测试。
- 在没有实现这条端到端链路前，把这些字段当作版本记录，不要承诺它们会出现在界面中。

### 3.5 图片与路径限制

- 支持 `.png`、`.jpg`、`.jpeg`、`.webp`；单张文件必须大于 0 且不超过 `16 MB`。
- 图片尺寸不能超过 `16384px` 的任一边，总像素不能超过 `50 MP`。
- 推荐首页和任务页均从 `16:9` 开始制作，例如 `2560×1440`；超宽图更适合 `banner`，而不是首页主视觉。
- 视差背景与前景应使用同一画幅、同一裁切基准。前景最好为带透明通道的 PNG/WebP，只保留可产生层次感的轮廓。
- 文件名保持 ASCII、小写、短横线分词，例如 `home-hero.png`；不要使用保留 Windows 名、绝对路径或 `..`。

## 4. “我想改 X，需要动哪里？”

| 目标 | 需要修改 | 通常不需要修改 | 必做验证 |
|---|---|---|---|
| 换一张首页主图 | `theme.json` 的 `image` 与图片文件 | 注入器、CSS | `-ValidateOnly`、首页截图 |
| 调整主体位置/文案避让 | `art.focusX`、`art.focusY`、`art.safeArea` | CSS | 宽/窄窗口截图 |
| 换任务页氛围图 | `taskImage`、`art.taskMode` | 首页图片 | 长对话可读性截图 |
| 新增视差 | 两张对应图、`backgroundImage`、`foregroundImage`、`art.parallax` | CSS 动画 | reduced motion、首屏与任务页检查 |
| 改主题的主强调色 | `palette.accent` | 全局 Angelina token | 浅/深焦点、选中与错误态 |
| 做一个新配色家族 | 外部主题包中的新目录与图片；必要时新 `appearance` 组合 | 已发布主题的稳定 ID | 主题切换与截图 |
| 改卡片、侧栏、输入框、菜单样式 | `engine/assets/dream-skin.css` | `theme.json` | CSS 语法、浅/深截图、交互检查 |
| 加首页装饰 DOM、时钟或状态标签 | `renderer-inject.js` 与 CSS | 单独的主题图片 | renderer 测试、真实 CDP 验证 |
| 让新的主题字段影响 UI | 注入器字段归一化、renderer 模板、CSS、测试 | 只改 JSON | payload 检查与热切换 |
| 适配 Codex DOM 更新 | renderer 选择器、CSS 选择器、QA 清单 | 主题图片 | 全套测试、真实截图 |
| 改启动器窗口或 `--launch` 行为 | 完整源码 checkout 的 GUI 项目 | 发布包中的 `.exe` | 重建 GUI、安装/启动回归 |

## 5. 制作一个新主题的标准流程

1. 从主题包仓库复制一个最接近的目录，或新建一个只包含 `theme.json` 和图片的主题目录。
2. 在 `theme.json` 中先改 `id`、`name`、`image`、`appearance` 和 `art`。已发布的稳定 `id` 不要复用。
3. 替换首页主图；需要任务图、视差时再逐项加入，先保证最小主题可用。
4. 以 `safeArea` 为依据，在 Codex 首屏实际窗口中检查文案、输入框与人物/高光是否冲突。
5. 运行静态校验：

   ```powershell
   powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\switch-theme.ps1 `
     -ThemeDirectory '.\themes\My Theme' -ValidateOnly

   node .\engine\scripts\injector.mjs --check-payload --theme-dir '.\themes\My Theme'
   ```

6. 用主题切换器导入/应用，或先启动带主题的 Codex 后进行热切换。分别检查首页、已有对话、项目选择、菜单、侧栏、右侧面板和输入框。
7. 保存浅色/深色截图和必要的提示词/素材来源，再更新 `CHANGELOG.md` 与 `engine/references/qa-inventory.md`。

## 6. 启动器与控制台说明

### 6.1 一个 EXE，两种入口

Release 中唯一有效的 GUI 程序是 `Codex自定义主题.exe`：

| 启动方式 | 模式 | 预期行为 |
|---|---|---|
| 直接运行、无参数 | 主题切换器/统一控制台 | 扫描主题、提供首页与任务页预览、导入和应用主题 |
| 带 `--launch` | 一键启动器 | 启动或唤醒当前主题对应的 Codex 会话，成功后自动关闭 |
| 桌面或开始菜单的“Codex 自定义主题” | 一键启动器 | 快捷方式传入 `--launch` |
| 开始菜单的“Codex 主题切换器” | 统一控制台 | 快捷方式不传参数 |

`Codex自定义主题启动器.exe.obsolete` 与 `CodexThemeSwitcher.exe.obsolete` 是旧名称留档，不能作为文档、快捷方式或发布入口。

### 6.2 一键启动器做什么

1. 读取活动主题和 `%LOCALAPPDATA%\CodexDreamSkin\state.json`。
2. 复核 watcher PID、端口与 Browser ID，并只访问 `127.0.0.1` 上的 CDP 端点。
3. 健康会话存在时，激活 Codex 窗口。
4. 会话缺失时调用 `engine/scripts/start-dream-skin.ps1`：启动或连接 Codex、启动 watcher、写入状态并验证注入。
5. 如果普通 Codex 已运行但没有 CDP，会请求允许重启，避免无提示丢失未发送输入。

健康唤醒通常约 `1.2 秒`；冷恢复可能约 `18 秒`，并取决于 Store 更新、机器性能与 Codex 启动时间。若启动器停留过久，优先查看 `%LOCALAPPDATA%\CodexDreamSkin\injector-error.log` 和 `verify.log`，不要反复强制结束进程。

### 6.3 统一控制台如何使用

- **选择主题**只改变当前选择；预览首页和任务页后再应用。
- **应用主题**先校验图片和 `theme.json`，将主题写入受管 `active-theme` 快照；CDP 会话可用时立即热切换，否则保存为下次启动主题。
- **启动 Codex**先保存当前选择，再执行一键启动流程。
- **导入主题**只接受顶层直接包含 `theme.json` 的目录，且会拒绝绝对路径、穿越路径和重解析点。
- 控制台最小化到系统托盘后，双击托盘图标恢复；设置可以改为保持窗口打开。

Release 根目录中的 `engine/`、`themes/` 和 `switch-theme.ps1` 是 GUI 的运行依赖。移动程序时必须整体移动目录，不能只复制 EXE。

## 7. 维护时还应记录什么

这份指南覆盖“如何做”，但每次发布还应维护下列事实记录：

| 文档/记录 | 何时更新 | 应记录的内容 |
|---|---|---|
| `README.md` | 用户路径或安装方式改变 | 安装、日常使用、FAQ、入口链接 |
| `README.en.md` | 对外英文说明改变 | 与中文 README 一致的用户可见行为 |
| `CHANGELOG.md` | 每次准备发布 | 用户可感知的新增、修复、兼容性和已知限制 |
| `engine/references/qa-inventory.md` | UI/DOM/截图验证完成 | 测试环境、覆盖页面、浅/深截图和已验证行为 |
| `engine/references/asset-provenance.md` | 新增或替换艺术资源 | 资源来源、许可、生成提示词和修改记录 |
| `engine/references/runtime-notes.md` | CDP、Store 或恢复策略改变 | 进程身份、状态文件、端口和兼容性说明 |

建议在后续版本增加一张“Codex 版本/DOM 兼容性矩阵”：记录验证日期、Codex 版本、关键选择器、已知失效区域和对应截图。它能在 Codex 更新后快速判断是主题素材问题还是 DOM 适配问题。

## 8. 验证与发布检查

每次修改后，从最小验证逐步扩大范围：

```powershell
# PowerShell 运行时与主题存储回归
powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\engine\tests\run-tests.ps1

# JavaScript 语法检查
node --check .\engine\scripts\injector.mjs
node --check .\engine\assets\renderer-inject.js

# 主题目录校验
powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\switch-theme.ps1 `
  -ThemeDirectory '.\themes\My Theme' -ValidateOnly
```

发布前人工检查：

- 至少导入一套浅色或深色主题，并验证其能够被扫描、预览、应用和重新启动。
- 首页、任务页、建议卡、输入框、菜单、左栏、右栏和调整大小手柄可读且可交互。
- `prefers-reduced-motion` 时没有视差或持续装饰动画。
- 暂停、恢复、热切换和冷启动不需要修改官方 Codex 文件。
- Release 同时包含 EXE、`engine/`、空的 `themes/` 导入目录、`switch-theme.ps1`、安装/卸载快捷方式脚本和本文档。

## 9. 常见问题的定位顺序

| 现象 | 优先检查 |
|---|---|
| 主题未出现在控制台 | 目录层级、`theme.json`、`id`、图片相对路径和图片格式 |
| 主题能预览但不能应用 | `switch-theme.ps1 -ValidateOnly` 输出，以及运行时 `active-theme` 权限 |
| 提示没有可连接会话 | 先用一键启动器；再检查 `state.json`、`injector-error.log` 和 Browser ID |
| 热切换后只部分生效 | `--check-payload`、watcher 日志、Codex DOM 是否更新；不要直接编辑 `active-theme` |
| 背景挡住文字或输入框 | `safeArea`、焦点坐标、主图裁切和 hero 遮罩；优先修素材/字段而非加厚蒙版 |
| 视差无效或画面跳动 | 两张视差图是否成对、`art.parallax` 是否为 `true`、reduced motion 是否开启、图片是否同画幅 |
| 启动器要求重启 | 当前 Codex 未带 CDP；先保存未发送内容，再授权重启或取消 |

## 10. 改动提交前的最小自检

- 我修改的是主题数据、渲染样式、注入逻辑还是 GUI？文件位置与职责是否匹配？
- 新主题是否有唯一稳定的 `id`，并且不覆盖已发布主题？
- 主题字段是否真的被运行时消费，而不是只写在 JSON 中？
- 动效是否克制、可停止、不会挡住交互，并尊重 reduced motion？
- 图片来源、授权与生成提示词是否已记录？
- README、CHANGELOG、QA 截图和兼容性说明是否需要同步更新？

只要这六个问题能明确回答，主题工作就不会只停留在“效果看起来不错”，而会成为可维护、可验证、可发布的产品改动。
